# Vyara OS — Slice 1 Build Spec (for Claude Code)

> **Read first, every session:** `CONSTITUTION.md` (architecture law) and `design.md` (UX/UI law). This spec scopes the first vertical slice. Build it completely and make it work end-to-end before anything else exists. Do **not** build toward the full system — build this slice.
>
> **Operating mode:** Work incrementally through the six steps below. After each step, ensure the app runs, then continue. Only pause for input when you hit a **genuinely blocking** decision — at most **3 blocking decisions + 5 recommendations** per step, then proceed with a stated assumption. Don't generate the whole system's design before coding; design just enough for the step you're on.

---

## The slice

**Lead → Project → Sample → Quote → Task → Timeline → Notification**, on **desktop** (office) and **mobile** (field engineer), built from scratch.

**The one capability that must feel magic:** when a specified project reaches the **paving stage**, the system automatically creates a follow-up task and notifies the owner — so won specs stop leaking. Everything else supports this.

**Pilot-grade, not production-hardened.** Real enough for a small group of Vyara engineers to use on real projects and for us to demo with confidence.

---

## In scope

- Auth + multi-tenant foundation (single tenant live, `tenant_id` everywhere) + 3 roles: `admin`, `sales_engineer`, `manager`.
- **Entities (minimal set — only what this slice needs):** `tenant`, `user`, `firm`, `contact`, `project`, `project_stakeholder`, `specification`, `product` (a simple catalog, ~10 seeded SKUs), `sample_request`, `quotation`, `quotation_line`, `task`, `activity` (timeline), `notification`, and a lightweight **`pipeline_stage`** config (data-driven, not hardcoded — see below).
- **Desktop screens:** Dashboard · Projects (list) · Project Detail (tabs: Overview, Stakeholders, Specifications, Samples, Quotes, Timeline, Tasks) · Tasks · Contacts.
- **Mobile (field) screens:** Today (tasks + paving-stage alert) · Projects · Project quick-view with quick actions (log visit, request sample, create quote, call) · Quick-add.
- **Flows:** create firm + contact → create project + record specification(s) → request sample + update status/outcome → create quote (line items, simple pricing) + send → advance stage → **paving-stage auto-task + notification** → complete tasks → timeline auto-logs every change.
- **Common spine on core objects** (per Constitution): Timeline, Tasks, Activities, Notifications, Audit. (Documents and Comments: light placeholders this slice.)
- **Field voice note → activity:** start with a text "quick note" that creates a structured activity; wire Sarvam STT as a fast-follow within the slice if time allows.
- **Quote send + reminders:** basic AiSensy WhatsApp send for a quote, and an Inngest scheduled job for the stale-quote nudge and the paving-stage check.

## Out of scope (do NOT build this slice)

Orders · dispatch · invoices · **collections** (Slice 2) · dealers/portal (Slice 3) · tenders · complaints · document OCR/intelligence · full workflow engine (guards/SLA/escalation) · pricing engine (use simple per-line pricing) · multiple pipeline segments beyond Architect + a generic default · form builder · production/Tally integration · advanced MIS/BI · React Native (use responsive PWA).

If something here feels needed, note it and move on — it belongs to a later slice.

## Minimal data model notes

- `project` has `segment` and a `stage` referencing data-driven `pipeline_stage` rows (seed the Architect stages: Specified → Tracking → Paving stage → Quoting → Won). **Don't hardcode stages in code**, but don't build the full transition/guard engine either — a simple ordered stage list + a `project_stage_history` row on each change is enough.
- `project_stakeholder` = N–N join (project × contact × role: specifier / buyer / influencer).
- `specification` links project × product (+ qty, finish) — this is what drives the paving-stage follow-up.
- `quotation` 1–N `quotation_line`; **snapshot price onto the line**.
- `task` is polymorphic-light: `(tenant_id, project_id?, type, title, due, priority, done, assignee)`. **Tasks can be generated by the system** (paving-stage, stale quote, sample-no-outcome) or created manually — per Constitution, "everything generates tasks."
- `activity` = the timeline feed (typed: created/assigned/stage_changed/sample/quote/note/ai/notification).
- Every table: `tenant_id`, audit columns, soft delete. Supabase RLS by `tenant_id` + territory/role. Margin/cost masked from `sales_engineer`.

## Stack & setup

Next.js (App Router) · Supabase (Mumbai — Postgres, Auth, RLS, Storage) · Tailwind + **shadcn/ui** themed from `design.md` tokens · lucide-react · react-hook-form + zod · TanStack Table · **Inngest** (scheduled checks, notifications) · AiSensy (WhatsApp, basic) · Vercel. Responsive/PWA for the field mobile (bottom-nav layout per `design.md` §7).

## Build sequence (six incremental steps)

**Step 1 — Foundation.** Scaffold Next.js + Supabase + Auth + `tenant_id` + RLS baseline. Set the `design.md` tokens as the shadcn theme (`globals.css` CSS vars + Tailwind config) before any screen. Build the app shell: desktop sidebar + topbar, mobile bottom-nav. Wire Inngest. *Done when: a logged-in user sees an empty themed shell on both layouts.*

**Step 2 — Data layer.** Migrations for the entity set + RLS policies + seed data (the Surat scenario: a township project, an architect, a contractor, ~10 SKUs, a couple of projects across stages). *Done when: seeded data is queryable under RLS.*

**Step 3 — Projects + Specifications.** Contacts/Firms (list + create), Projects (list + create), Project Detail shell with the data-driven stage stepper, Specifications tab. *Done when: you can create a firm, contact, project, and record a specification.*

**Step 4 — The spine + the hero.** Tasks (manual + system-generated), Activities/Timeline (auto-logged on every change), Notifications. The **paving-stage trigger**: on stage change to "Paving stage" (and via an Inngest daily check), auto-create the follow-up task + notification. *Done when: advancing a project to paving stage spawns the task and it shows on the dashboard and mobile Today.*

**Step 5 — Samples + Quotes.** Sample request + status/outcome; Quote builder (line items, simple pricing, snapshot) + send via AiSensy (basic) + stale-quote Inngest nudge. *Done when: you can send a quote and see a reminder fire.*

**Step 6 — Field mobile + dashboard + demo polish.** Mobile Today/Projects/quick-view/quick-add; voice/quick-note → activity; dashboard (pipeline by stage, today's follow-ups, the paving-stage alert). Empty/loading/error states everywhere per `design.md`. *Done when: a field engineer can run the full flow on a phone and a manager sees it on the dashboard — a demoable pilot.*

---

## Definition of done for Slice 1

A pilot user can, end to end: capture a contact → create a project → record a specification → request and track a sample → raise and send a quote → and be **automatically reminded to follow up when the project hits paving stage** — on desktop and on a phone, with a live timeline and dashboard. That's the slice. Ship it, put it in front of Vyara, learn, then start Slice 2 (Order → Dispatch → Invoice → Collection).
